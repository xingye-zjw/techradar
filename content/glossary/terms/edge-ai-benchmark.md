# 边缘AI基准测试

**边缘 AI 基准测试（Edge AI Benchmarking）** 是一套标准化评估边缘端硬件（手机 SoC、边缘 GPU、NPU、FPGA、微控制器 MCU）在 AI 推理任务上性能的方法体系。它解决的核心痛点是：**硬件厂商标称的「XX TOPS 算力」和实际业务跑模型的速度/功耗/精度，相关性往往不到 50%**。做基准测试就是把「宣传 TOPS」还原成「业务能用的性能」。

## 为什么不能只看厂商 TOPS？

```
厂商宣传：某 NPU 算力 20 TOPS（每秒 20 万亿次运算）
实际跑 YOLOv8s 640×640：23 FPS
换算有效算力：23 × 模型 FLOPs(18.2G) = 419 GOPS ≈ 0.4 TOPS

利用率 = 0.4 / 20 = 2%
```

为什么差 50 倍？因为：

1. **算子支持度差**：20 TOPS 只算 INT8 Conv3×3，但模型里有大量 SiLU/Sigmoid/Resize/Concat 这些「非典型 NN 算子」，跑在通用 CPU 上，慢死
2. **内存带宽瓶颈**：算力够，但 DDR 喂不饱数据，计算单元 98% 时间在等数据（典型的 Memory-Bound 模型）
3. **编译优化不到位**：模型从 PyTorch → ONNX → 厂商 IR 转换，大量算子融合（Conv+BN+ReLU 合 1 个）、量化校准失败，有效指令少
4. **散热降频**：跑 2 分钟后芯片结温超过 105°C → 自动降频到 50%，厂商 TOPS 是散热片全开峰值跑分

基准测试的意义就是：**抛开厂商纸面参数，统一标准、统一模型、统一环境，测出真实可落地的性能**。

## 主流基准测试套件

### 1. MLPerf 系列（业界事实标准，ML Commons 组织）

MLPerf 是 Google、Meta、NVIDIA、Arm 等 80+ 公司联合制定的，分四个子套件：

| 套件                              | 目标硬件                                  | 评测任务                                                                | 关键指标                                    |
| --------------------------------- | ----------------------------------------- | ----------------------------------------------------------------------- | ------------------------------------------- |
| **MLPerf Training**               | 云端 GPU/TPU 集群                         | BERT、GPT-3、ResNet、U-Net 等 8 个模型训练到 SOTA 精度耗时              | 训练时间（分钟）、功耗（kWh）、成本（$）    |
| **MLPerf Inference - Datacenter** | 服务器级 GPU/CPU                          | ResNet50、BERT、DLRM、3D-UNet、GPT-J 6B、Stable Diffusion 离线+在线推理 | 离线吞吐量 Samples/sec、在线 P50/P99 延迟   |
| **MLPerf Inference - Edge**       | 手机 SoC、桌面 x86、Jetson、嵌入式        | 同 Datacenter 任务 + MobileNet + 语音 RNN-T                             | 单样本延迟 ms、系统功耗 W                   |
| **MLPerf Tiny**                   | **Cortex-M/RISC-V MCU 级**（TinyML 场景） | KWS 唤醒词、VWW 视觉唤醒词、IC 图像分类、AD 异常检测 4 个 tiny 模型     | Latency ms、Accuracy %、Energy μJ/inference |

**MLPerf 的核心原则（保证公平）**：

- **Closed Division（强制参加）**：所有人必须用组委会指定的模型权重、预处理、精度要求（INT8/FP16），不能改结构，只能做编译/部署优化——比的就是硬实力
- **Open Division（可选）**：允许改模型、改精度，测的是「厂商软硬件极限」
- **Power 赛道（强制接功耗仪）**：每款板子用 YOKOGAWA 或 Monsoon 功耗仪接 12V 输入，精确测每 inference 的 mJ

## 边缘 AI 基准的四大核心指标

任何边缘 AI 基准，都必须同时测这 4 个维度，缺一个就不完整：

### 1. 延迟 Latency（ms / inference）

端侧最在意的指标——自动驾驶感知要求 **端到端 < 50 ms**，语音唤醒 < 300 ms。

- **Cold Latency**：模型第一次加载 + 推理（含动态库加载、权重复位、CPU cache miss）→ 影响 App 冷启动体验
- **Warm Latency**：连续跑 100 次取 P50/P90/P99 → 真实生产指标（**P99 才是关键！**）
- **Batch Latency**：批处理 N=4/8 时单样本平均延迟 → 衡量吞吐能力

### 2. 吞吐量 Throughput（FPS / QPS / Samples/sec）

能并行的场景（视频流 16 路分析）看 throughput。
注意：**延迟和吞吐量是 trade-off 关系**，通常要同时给「N=1 延迟」和「满 batch 吞吐量」两条曲线。

### 3. 精度 Accuracy（相对于浮点基线）

INT8/INT4 量化不能把精度掉太多：

- 分类：Top-1 掉点 < 1% 可接受
- 检测：mAP 掉点 < 2% 可接受
- NLP：F1 掉点 < 3% 可接受

精度损失大的「高性能」都是耍流氓。

### 4. 能效比 Energy Efficiency（mJ / inference 或 FPS/W）

边缘设备是电池供电的核心指标。

| 硬件                   | YOLOv8s 640×640 能效比                       | 典型功耗    |
| ---------------------- | -------------------------------------------- | ----------- |
| NVIDIA RTX 4090        | ~0.8 mJ/inference（1000+ FPS / 450W）        | 450W        |
| NVIDIA Jetson AGX Orin | ~3 mJ/inference（120 FPS / 60W）             | 15~60W 可调 |
| 高通骁龙 8 Gen 3 NPU   | ~6 mJ/inference（60 FPS / 4W）               | 3~6W        |
| STM32H7 Cortex-M7      | ~60 mJ/inference（MobileNetv2 2 FPS / 0.3W） | 0.3W        |

> 用户关心的是：**同精度下，1 焦耳电能跑多少帧推理**，这个指标直接决定设备电池续航。

## 企业内部基准测试最佳实践

官方基准套件是「通用任务」，但企业有自己的「业务模型」——必须双轨并行。

### 第一步：定义自研基准（Benchmark Suite）

至少选 5 个「真实业务模型」：

```python
BENCHMARK_MODELS = [
    # (名称, onnx路径, 输入shape, 预处理, 测试集, 精度阈值)
    ("yolov8s_det", "yolov8s.onnx", (1,3,640,640), letterbox,
     val_2000_img, 0.50),                         # mAP@0.5 ≥ 50%
    ("unet_seg", "unet_industrial.onnx", (1,3,512,512), normalize,
     val_seg, 0.88),                              # mIoU ≥ 88%
    ("asr_conformer", "conformer_int8.onnx", (1,80,3000), fbank,
     test_clean_100h, 0.92),                      # CER ≤ 8%
    ("bevformer_tiny", "bev.onnx", (6,3,480,640), bev_preprocess,
     val_nuscenes_500, 0.45),                     # NDS ≥ 45%
]
```

### 第二步：统一测试环境规范

```yaml
环境清单（必须记录到测试报告附录）：
  硬件:
    型号: Jetson Orin NX 16GB
    电源模式: MAXN (15W) / MODE_15W / MODE_10W 三套都测
    散热: 官方风扇 + 散热片，室温 25°C（必须加热电偶测芯片结温）
    存储: Samsung 980 Pro 1TB NVMe
  软件:
    JetPack: 5.1.2
    CUDA: 11.4.315, TensorRT: 8.5.2, cuDNN: 8.6.0
    内核: 36.2.0-tegra, CPU 调度策略: performance
    内存锁定: 把模型权重 mlock 到 RAM，禁止 swap
  测试方法:
    预热: 前 100 次不计入统计
    样本数: 每种模型连续跑 2000 次 inference
    采样频率: 功耗仪 1kHz 同步记录电压电流
    输出统计: P50 / P90 / P99 / P99.9 延迟
```

## 常见陷阱与规避

### 陷阱 1：厂商演示视频「跑通」不等于「生产能用」

厂商演示板跑 MobileNetv2 99 FPS → 实际 YOLOv8s 只有 18 FPS → 为什么？因为前者 100% 算子都硬件加速了，后者一半算子 fallback CPU。
→ **规避方法：必须跑自己的业务模型，不能跑 demo 模型**。

### 陷阱 2：「峰值性能」≠「长时间稳定性能」

跑前 60 秒 100 FPS → 2 分钟后降到 45 FPS → 结温 110°C 过热降频了。
→ **规避方法：每款板子至少连续烤机 30 分钟，测「热稳定性能」而不是冷启动峰值。**

### 陷阱 3：PTQ 量化精度造假

厂商用 ImageNet 验证集里挑了 100 张「容易分」的猫图测精度 → 98% → 实际你业务数据测只有 89%。
→ **规避方法：精度测试必须用业务场景下的独立验证集，必须和浮点基线逐样本对齐统计误差。**

### 陷阱 4：只测单模型延迟，不考虑「端到端流水线」

真实业务流水线：

```
摄像头采集 → 预处理（Resize+归一化，10ms）→ NPU 推理（12ms）→ 后处理（NMS+解码，8ms） → 结果上传
                                   ↑
                    厂商只给你测了这一段 12ms → 但你实际 E2E = 30ms
```

→ **规避方法：必须测端到端全流程，前处理后处理都是要花时间的！**

## 基准测试自动化框架搭建

成熟公司会搭建内部「Edge AI CI 基准平台」：

```
┌───────────────────────────────────────────────────────────┐
│ Git 提交 → 模型变更/编译优化变更触发 CI                     │
└───────────────────────┬───────────────────────────────────┘
                        ▼
┌───────────────────────────────────────────────────────────┐
│ Benchmark Runner Server（K8s Job）                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │  Jetson Orin  │  │  RK3588 板子 │  │  骁龙开发板  │      │
│  │   Cluster    │  │   Cluster    │  │   Cluster    │      │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘      │
└─────────┼─────────────────┼─────────────────┼──────────────┘
          ▼                 ▼                 ▼
┌───────────────────────────────────────────────────────────┐
│ 统一采集器：                                                │
│  • 模型转换 + 精度验证（和黄金浮点版比）                     │
│  • Perf 事件 + 硬件 PMU 计数器（指令数、cache miss、访存 BW）│
│  • 功耗仪同步采样（mW 级 1kHz）                              │
│  • 结温 + CPU/GPU/NPU 利用率                                 │
└───────────────────────┬───────────────────────────────────┘
                        ▼
┌───────────────────────────────────────────────────────────┐
│ 基准报告入库（TimescaleDB + Grafana）                        │
│  • 每提交一个 commit 画一条性能趋势线                         │
│  • 任何模型/版本相对基准 regression > 3% → 自动发邮件报警     │
└───────────────────────────────────────────────────────────┘
```

> 这套系统是「AI 工程化的仪表盘」——没有它，你永远不知道某次代码改动对端侧性能的真实影响。

## 未来趋势

1. **LLM 边缘基准（MLPerf LLM Edge）**：2025 起重点测 Phi-3、Qwen1.5-1.8B、Llama-3-8B 在手机和 Jetson 上的推理速度 + 每 token 功耗 + 困惑度。
2. **存内计算（CIM / PIM）基准**：ReRAM、SRAM-CIM 新架构芯片出来，需要专门的基准去测「真实能效比」，而不是厂商宣传的 1000 TOPS/W。
3. **统一多模态边缘基准**：一套流水线同时测 CV 检测 + ASR + TTS + VLM 图文问答，模拟真实 AI 眼镜/机器人场景。
4. **碳足迹基准（Carbon-Performance）**：不再只测 mJ，而是把「某硬件一年推理 100 万次的等效碳排放 kgCO₂e」写进采购决策指标，双碳目标落地。

边缘 AI 基准测试本质是「硬件采购和工程优化的科学方法论」——它把「我觉得这个芯片可能快」变成了「我测过 14 个指标，这款芯片在我们的 5 个业务模型上综合最优，能效比第 2 名好 37%，精度掉点 < 1%，结温 82°C 稳定」的决策依据。在 AI 芯片百花齐放的今天，基准测试能力就是 AI 工程团队的「技术决策硬实力」。
