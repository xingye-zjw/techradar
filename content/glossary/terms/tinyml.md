# 微型机器学习

**TinyML（微型机器学习）** 是机器学习的一个子领域，专注于在资源极度受限的 **微控制器（Microcontroller Unit, MCU）** 和嵌入式系统上运行模型推理。典型硬件约束：RAM ≤ 512KB、Flash ≤ 2MB、算力 ≤ 1 GOPS、功耗 ≤ 1mW（纽扣电池供电可续航 1 年以上）。TinyML 的口号是 **「把 AI 塞进传感器里」**——让温湿度计、麦克风、三轴加速度计这些最底层设备不再只是采集和上传数据，而是本地就能做智能决策。

## TinyML vs 边缘 AI vs 云计算

| 维度         | 云计算（云端推理）                | 边缘 AI（Jetson/手机/NPU）           | TinyML（MCU 级）                             |
| ------------ | --------------------------------- | ------------------------------------ | -------------------------------------------- |
| **典型硬件** | A100/H100 服务器                  | Jetson Orin、骁龙 8 Gen、苹果 M 系列 | STM32H7、ESP32-S3、Cortex-M0/M4、RISC-V GD32 |
| **典型功耗** | 千瓦级（kW）                      | 10~60 瓦（W）                        | 微瓦~毫瓦（μW ~ mW）                         |
| **模型大小** | 数百 GB（FP16）                   | 数 GB ~ 数十 GB                      | 数 KB ~ 数 MB（INT8）                        |
| **典型延迟** | 数百 ms ~ 数秒（加网络）          | 10~500 ms                            | < 10 ms（本地硬实时）                        |
| **隐私性**   | 数据上传云端，有泄露风险          | 本地处理，较安全                     | 传感器端处理，0 数据出设备（最安全）         |
| **离线能力** | 断网即瘫                          | 可离线（依赖本地模型）               | 完全离线，无通信模块也行                     |
| **代表场景** | LLM API、图像搜索、海量数据批处理 | 自动驾驶感知、手机 AR、安防 NVR      | 振动异常检测、语音唤醒词、可穿戴心率预警     |

> TinyML 不是「边缘 AI 的缩小版」，而是**完全不同的设计范式**：边缘 AI 仍在追求「尽可能大的模型跑快点」，TinyML 追求「尽可能够用的模型塞进去、耗尽可能少的电」。

## 典型硬件生态

**三大主流 MCU 家族：**

| 厂商                     | 系列                | 核心                       | 典型频率    | RAM / Flash               | TinyML 友好度                             |
| ------------------------ | ------------------- | -------------------------- | ----------- | ------------------------- | ----------------------------------------- |
| **ST（意法半导体）**     | STM32L4/L5/U5/H7    | Cortex-M4/M7/M33           | 80~480 MHz  | 128KB~~1MB / 512KB~~2MB   | ⭐⭐⭐⭐⭐ 生态最成熟，Cube.AI 工具链完善 |
| **Espressif（乐鑫）**    | ESP32-S3 / ESP32-P4 | Xtensa LX7 / RISC-V        | 240~360 MHz | 512KB / 8MB（PSRAM 扩展） | ⭐⭐⭐⭐ WiFi/BLE 自带，AIoT 首选，便宜   |
| **Nordic（北欧半导体）** | nRF52840 / nRF5340  | Cortex-M4 / M33 + M33 双核 | 64/128 MHz  | 256KB / 1MB               | ⭐⭐⭐⭐⭐ 蓝牙低功耗顶级，可穿戴设备首选 |

**专用 TinyML 加速器（新秀）：**

- **Syntiant NDP120**：专门做语音唤醒 + 传感器融合，28nm 工艺，128KB SRAM，推理功耗仅 9μW
- **MAX78000（ADI）**：CNN 硬件加速器核 + 2 个 Cortex-M4F，80MHz，跑 MobileNetv1 仅 1.3ms
- **GAP9（GreenWaves）**：9 核 RISC-V 集群 + NN 加速器，做图像分类/音频降噪强手

## TinyML 全流程工具链

从训练到 MCU 部署，TinyML 有一整套「瘦身」流水线：

```
┌──────────────────────────────────────────────────────────────┐
│  步骤 1：数据集采集（TinyML 数据和 CV/NLP 完全不同）            │
│                                                                 │
│  • 振动异常检测：贴传感器在电机上采样 256Hz 三轴加速度 → 10 秒   │
│    一段样本 → 正常 / 不平衡 / 轴承磨损 / 齿轮断齿 四类           │
│  • 唤醒词（KWS，Keyword Spotting）：录制 100 人 × 50 遍         │
│    「小度小度」+ 100 小时负样本（背景噪音、日常对话 + 1000 个     │
│    其他易混淆词）                                                │
│  • 数据必须在目标硬件的真实传感器上采！电脑仿真麦克风/加速度计    │
│    和真实传感器响应差异 → Sim2Real 严重，直接部署准确率掉 20%+    │
└────────────────────────────┬─────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  步骤 2：轻量模型架构设计（不是把大模型量化就行，得从头选对模型）  │
│                                                                 │
│  ┌──────────────────────┬─────────────────────────────────────┐│
│  │ 任务类型              │ 推荐架构                             ││
│  ├──────────────────────┼─────────────────────────────────────┤│
│  │ 传感器时间序列分类    │ 1D CNN（3 层 Conv1D + 2 层 Dense）  ││
│  │                      │ 或 TCN（时域卷积网络）                ││
│  ├──────────────────────┼─────────────────────────────────────┤│
│  │ 唤醒词 / 小词汇量ASR  │ DS-CNN（Depthwise Separable CNN）   ││
│  │                      │ 或 BC-ResNet（残差连接版）           ││
│  ├──────────────────────┼─────────────────────────────────────┤│
│  │ 低分辨率图像分类      │ MobileNetv1/v2（宽度因子 0.25 版）   ││
│  │                      │ 或 EfficientNet-Lite0                ││
│  ├──────────────────────┼─────────────────────────────────────┤│
│  │ 异常检测（无标签）    │ 小 Autoencoder / IForest + 阈值      ││
│  └──────────────────────┴─────────────────────────────────────┘│
│  经验法则：模型参数量 ≤ Flash / 4（因为还要存权重 + 代码 + 其他）│
│            推理时激活峰值内存 ≤ 80% RAM（留 stack/heap 余量）   │
└────────────────────────────┬─────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  步骤 3：训练 + 瘦身三件套（QAT 量化 + 剪枝 + 知识蒸馏）         │
│                                                                 │
│  ① QAT（Quantization-Aware Training，量化感知训练）             │
│     在训练图中插入 FakeQuant 节点，模拟 INT8 量化误差 →          │
│     模型学会在量化噪声下也能工作 → 最终准确率掉 < 1%             │
│     （PTQ 后训练量化会掉 5~10%，不推荐对 < 1MB 小模型用）        │
│                                                                 │
│  ② 结构化剪枝（L1 Pruning + 正则化）                             │
│     先训一个稍宽的模型（如 64 通道 Conv）→ 把 L1 权重小的通道剪   │
│     掉 → 变成 32 通道 → 再微调 20 epoch → 精度几乎不变，         │
│     模型大小 -50%，推理速度 +30%                                │
│                                                                 │
│  ③ 知识蒸馏（Distillation）                                     │
│     训一个大老师模型（手机上能跑的 MobileNetv2 1.0）→ 软标签    │
│     蒸馏给 MCU 上的小学生 Tiny 模型 → 小模型准确率 +5~8 个点     │
└────────────────────────────┬─────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  步骤 4：转换 + 部署（三大推理框架任选）                         │
│                                                                 │
│  ┌─────────────────────────┬─────────────┬────────────────────┐│
│  │ 框架                    │ 出品方        │ 特点               ││
│  ├─────────────────────────┼─────────────┼────────────────────┤│
│  │ TFLite Micro            │ Google        │ 生态最成熟，示例最多││
│  │ ONNX Runtime Micro      │ Microsoft     │ 兼容 ONNX 全生态   ││
│  │ STM32Cube.AI            │ ST 意法半导体  │ 硬件优化最深，自动 ││
│  │                         │              │ 调内存分配，速度最快││
│  │ nnom / ELMA             │ 开源社区      │ 极轻量，无依赖，    ││
│  │                         │              │ 移植简单           ││
│  └─────────────────────────┴─────────────┴────────────────────┘│
│                                                                 │
│  转换输出：模型从 .h5 / .onnx → 一个 .c 数组 + .h 头文件：        │
│  const unsigned char model_data[] = { 0x00, 0x01, ... };         │
│  → 直接和业务代码一起交叉编译 → 刷进 MCU Flash                   │
└──────────────────────────────────────────────────────────────┘
```

## 经典任务 1：电机振动异常检测（落地最广泛）

工业场景：工厂里 1 万台电机，人工巡检每月一次 → 很多故障发现时已经烧毁。装振动传感器 + TinyML 做 24h 实时预警。

### 数据采集与预处理

```cpp
// STM32 每 4ms 读一次 ICM20689 三轴加速度计
// 采样率 250Hz → 10 秒一个样本 = 2500 点 × 3 轴 = 7500 个 int16

// 特征工程（不是原始波形直接喂模型！）
// 从 2500 点里提取 40 维时域/频域特征：
float features[40] = {
    // 时域：
    mean_x, std_x, peak_x, rms_x, kurtosis_x, skewness_x, crest_x,
    mean_y, std_y, peak_y, rms_y, kurtosis_y, skewness_y, crest_y,
    mean_z, std_z, peak_z, rms_z, kurtosis_z, skewness_z, crest_z,
    // 频域（先做 256 点 FFT → 得到频谱）：
    band_0_25hz_energy,   band_25_50hz_energy,
    band_50_100hz_energy, band_100_250hz_energy,
    spectral_centroid,    spectral_rolloff,
    top_3_harmonic_freq,  ...
};
```

### 模型与部署

```
模型：3 层 Conv1D（8/16/32 通道，3×1 核）+ 2 层 Dense (64 → 4)
参数量：24KB
训练准确率（四类）：97.6%
QAT INT8 量化后掉点：0.8% → 96.8%
STM32L4 80MHz 推理时间：2.3ms，推理一次功耗：~3μAh
→ 用 240mAh 纽扣电池，每 5 分钟测一次 → 续航 3 年
```

## 经典任务 2：语音唤醒词 KWS（出货量最大）

智能音箱/蓝牙耳机/车机的「嘿 Siri / 小爱同学」——全都是 TinyML KWS。

要求指标：

- **唤醒率（Recall）> 98%**：100 次真实喊「小爱同学」，至少 98 次被唤醒
- **误唤醒率（False Accept）< 1 次 / 24 小时**：看电视聊天一整天，不能自己突然答应
- **端到端延迟 < 300ms**：人喊完到设备响应，体感必须「立刻」

### 数据流水线

```
音频输入（16kHz 单声道，PCM 16bit）
   ↓
  25ms 帧长 × 10ms 帧移 → 每帧 400 采样点
   ↓
  FBank（40 维梅尔滤波器组）→ 得到 40×N 特征图（N = 秒数 × 100）
   ↓
  DS-CNN（3×3 Depthwise Conv × 4 层 + 平均池化 + Softmax 12 类）
   ↓
  输出 12 类概率：
    [静音, 噪音, 小度小度, 小度小度+噪声1, ... 其他词...]
   ↓
  连续 10 帧（100ms）「小度小度」概率均 > 0.7 → 触发唤醒中断
```

### 硬件上的极致优化

唤醒词是常开启任务（Always-On），必须让 MCU 在 DeepSleep 模式下仅留一个低功耗外设看门运行：

- nRF5340 用 Network 核（64MHz Cortex-M33，供电 0.8V）跑 KWS，平均电流 **< 100μA**
- 真正触发唤醒后，才唤醒 App Core（128MHz）+ WiFi/BLE 模块去做云端 ASR → 整体续航提升 10 倍

## 经典任务 3：可穿戴 ECG 心律不齐识别

智能手表/运动手环里的医疗级心率检测，实时看 ECG 波形，在手表本地识别房颤（AFib）、室早（PVC）等。

模型：用 LSTM + Attention 或者 1D ResNet，输入 30 秒 250Hz ECG = 7500 采样点 → 输出 1 份心律报告。Flash 占用 < 512KB，在 Apollo4 Blue Plus 上 13.7ms 推完。

关键价值：**本地识别后只把异常片段上传**，否则连续心电 24h 有 2.16 亿采样点 → 蓝牙/4G 传不动。TinyML 让「只传 1% 异常数据」，隐私性也大幅提升。

## TinyML 的 4 个常见坑（踩过血泪史）

### 坑 1：浮点 vs 定点精度差异

训练时是 FP32，部署时 INT8，实际部署发现类别概率分布都偏移了？
→ **必须做 QAT（训练感知量化）+ 用真实硬件推理结果做校准集**，不是简单 PTQ 了事。

### 坑 2：传感器响应不匹配

在开发板上连仿真信号源测试 99% 准确率，装到真实电机上掉到 80%？
→ **训练 + 测试集必须用目标硬件同款传感器、同安装方式、真实场景采集**，千万别省这一步。

### 坑 3：堆内存爆栈

TensorArena（推理工作内存）算小了，跑着跑着 HardFault_Handler 死机？
→ 离线算 peak 内存 × 1.3 留余量，实际跑时用硬件 BKP 寄存器监测 stack/heap 水位线。

### 坑 4：温度/电压漂移

室温 25°C 正常，跑到 70°C 工业现场准确率崩？
→ 训练时加传感器数据温度梯度增强（高低温箱采样）+ 权重和激活做对称量化，抗漂移。

## 未来方向

1. **Tiny Transformer for MCU**：MobileViT 等带局部注意力的轻量 Transformer 架构 + INT4 量化，2MB Flash 就能跑，精度比纯 CNN 高 3~5 个点。
2. **Tiny 多模态融合**：同时输入加速度 + 麦克风 + 温度，做 3 模态传感器融合，识别「设备真故障 vs 被人碰了一下」。
3. **TinyLLM 降临**：1.5B 参数的 Phi-3 Mini INT4 量化后约 800MB——现在塞不进 MCU，但下一代存内计算（CIM）芯片（兆易创新 RISC-V + ReRAM）出来后，几年内就能在「大 MCU」上跑 100M 级的 TinyLLM，让传感器设备具备本地自然语言对话能力。
4. **联邦 TinyML（Federated Learning on Edge）**：10 万台设备各跑各的 TinyML，把本地梯度的差分隐私版上传云端聚合 → 中央模型每周下发更新，**原始数据永远不离开设备**，彻底解决数据合规。

TinyML 是 AI 在「端侧」的最后一公里。它不像大模型那样上头条，但它是让 AI 真正物理嵌入现实世界的每一个螺丝、每一个电机、每一件衣服、每一颗传感器的核心技术。高通预测到 2028 年，全球会有 **超过 40 亿台 TinyML 设备**在持续运行——这是一个比云端 LLM 大得多的物理市场。
