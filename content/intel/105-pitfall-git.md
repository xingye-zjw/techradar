## Git 合并冲突处理不当导致代码丢失

// 快速修复

合并前先 `git stash` → 解决冲突后 `git add` + `commit` → `git stash pop`

// 现象表现

- × git merge 后某些文件变成旧版本
- × git pull 后本地修改被覆盖
- × 冲突文件内容混乱，难以判断正确版本

// 排查步骤

- 01 合并前先提交本地修改，确保工作区干净
- 02 解决冲突时仔细理解每处变更，不要盲目选择某一侧
- 03 永远不要在未提交时做 pull/merge 操作
- 04 冲突解决后使用 `git status` 确认所有文件已正确标记

## 误提交大文件/敏感信息到 Git 历史

// 快速修复

使用 `git filter-branch` 或 BFG Repo-Cleaner 清理历史 → 确认清理完成后重置仓库

// 现象表现

- × 仓库体积过大（超过正常项目体积数倍）
- × git clone 极慢或超时
- × 敏感信息（密码、密钥、Token）泄露
- × .git 目录占用大量磁盘空间

// 排查步骤

- 01 用 `git lfs` 管理大文件（二进制文件、视频、图片等）
- 02 定期检查仓库大小 `du -sh .git`，异常时及时排查
- 03 提交前检查 .gitignore 配置文件是否完善
- 04 使用 `git log --diff-filter=A --summary` 查找历史上的大文件提交
- 05 敏感信息泄露后立即重置相关凭证并清理历史

## 分支命名混乱导致协作困难

// 快速修复

采用一致的分支命名规范：`feature/xxx`、`bugfix/xxx`、`release/xxx`、`hotfix/xxx`

// 现象表现

- × 不知道哪些分支在做什么
- × 合并错了分支
- × 过期分支未清理，仓库存在大量僵尸分支
- × 多人基于同一功能创建了重复分支

// 排查步骤

- 01 制定并文档化分支命名规范，全团队遵循
- 02 使用 GitFlow 或类似的工作流程管理分支生命周期
- 03 定期清理已合并的分支 `git branch --merged | xargs git branch -d`
- 04 使用 `git branch -vv` 检查分支跟踪状态和远程情况
- 05 在项目 README 或 CONTRIBUTING 文件中明确分支策略

## rebase 后强制推送导致同事代码丢失

// 快速修复

永远不要强制推送已推送的提交 → 使用 `git pull --rebase` 而非 merge

// 现象表现

- × 同事的提交消失，只剩下自己的提交
- × 本地分支与远程分叉，无法正常推送
- × 强制推送后代码无法恢复，只能从头再来
- × 多人协作时产生难以解决的冲突循环

// 排查步骤

- 01 避免强制推送，除非明确知道远程仓库没有其他人基于该提交工作
- 02 优先使用 `pull --rebase` 保持历史线性，而不是 `merge`
- 03 定期 `git fetch` 同步远程状态，提前发现分叉
- 04 如需更新远程分支，先与同事沟通，确认无人基于该分支工作
- 05 在团队规范中明确禁止对主分支进行强制推送
