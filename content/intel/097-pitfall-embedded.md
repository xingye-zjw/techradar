## C语言指针越界访问导致段错误

// 快速修复

Valgrind检测 + 安全字符串函数（strncpy/snprintf）+ 指针释放后置NULL + 嵌入式启用MPU限制访问范围

// 现象表现

- × Segmentation fault (core dumped)
- × 程序崩溃在看似无关的代码位置
- × 数据被莫名其妙修改
- × 嵌入式系统死机或看门狗复位

// 排查步骤

- 01 使用Valgrind检测内存错误（PC端开发时）
- 02 数组/指针访问前检查索引范围和NULL
- 03 字符串操作用strncpy替代strcpy/sprintf，并确保字符串以'\0'结尾
- 04 指针释放后立即设为NULL，防止悬空指针
- 05 嵌入式系统启用MPU限制关键内存区域访问范围
- 06 编译时启用-fsanitize=address定位内存问题

## RTOS任务栈溢出

// 快速修复

增大栈大小 + 监控高水位标记(uxTaskGetStackHighWaterMark) + 避免大局部变量 + 启用栈溢出检测

// 现象表现

- × HardFault_Handler被触发
- × 任务运行一段时间后崩溃
- × 变量值被莫名其妙修改
- × 调试时发现LR(链接寄存器)值异常

// 排查步骤

- 01 使用FreeRTOS的uxTaskGetStackHighWaterMark()监控栈使用情况
- 02 任务初始化时设置较大栈空间（建议至少2KB，留20-30%余量）
- 03 避免在栈上分配大局部数组/结构体，改为静态或全局分配
- 04 减少函数调用嵌套深度，优化递归调用
- 05 启用 FreeRTOS 软件栈溢出检测：configCHECK_FOR_STACK_OVERFLOW=2（基于栈哨兵检测）；MPU 硬件栈保护需单独配置 MPU 区域
- 06 使用静态分析工具检查函数最大栈深度

## GPIO配置错误导致外设不工作

// 快速修复

确认引脚功能复用 + 检查时钟使能 + 验证GPIO模式(输入/输出/复用) + 验证电平匹配(3.3V/5V)

// 现象表现

- × 外设无输出或无输入
- × 配置了但信号不对
- × 烧录后程序完全不运行
- × 引脚电平不稳定或值不符合预期

// 排查步骤

- 01 检查GPIO引脚功能复用配置，确认使用了正确的AF(Alternate Function)
- 02 确认外设时钟已使能（如__HAL_RCC_GPIOX_CLK_ENABLE()）
- 03 检查GPIO模式配置：输入(IN)、输出(OUT)、复用(AF_PP/AF_OD)
- 04 验证电平匹配：3.3V系统不要接5V外设，或使用电平转换电路
- 05 检查推挽/开漏模式选择，开漏需外接上拉电阻
- 06 使用示波器或逻辑分析仪观察实际信号波形
- 07 检查PCB布线：走线长度、阻抗匹配、信号完整性

## 串口通信乱码或数据丢失

// 快速修复

匹配两端波特率 + 检查流控(RTS/CTS)配置 + 验证电平转换(RS232/TTL) + 确认校验位和数据位

// 现象表现

- × 接收数据乱码，字符显示为奇怪符号
- × 部分数据丢失，接收不完整
- × 偶尔能通信但不稳定，时好时坏
- × 发送端正常但接收端无响应

// 排查步骤

- 01 确认两端波特率精确匹配，建议使用示波器测量实际波特率
- 02 检查流控配置：RTS/CTS硬件流控或XON/XOFF软件流控
- 03 验证电平转换：TTL电平(0-3.3V/5V)与RS232电平(±12V)不可直接相连
- 04 检查校验位(Parity)和数据位设置：8N1/8E1/8O1等
- 05 使用示波器或逻辑分析仪观察TX/RX信号质量
- 06 检查串口地线连接，共地是通信正常的前提
- 07 确认UART时钟分频配置正确，外设时钟源稳定
