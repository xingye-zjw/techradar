## 贝叶斯优化搜索空间过大导致效率低下

// 快速修复

缩小搜索空间 + 先粗搜再精搜 + 基于先验知识设置范围

// 现象表现

- × AutoML跑了一天还没找到好模型
- × 每次试验结果都差不多，没有明显提升
- × 搜索完全是随机的，贝叶斯优化没有发挥作用
- × 高维空间中采样稀疏，优质区域难以被探索到

// 排查步骤

- 01 检查搜索空间维度和每个参数的取值范围是否合理，优先减少连续型参数的范围
- 02 先用网格搜索或随机搜索做少量试验，确定各参数的大致有效范围
- 03 基于领域经验和文献参考，将搜索空间缩小到合理区间
- 04 采用两阶段策略：第一阶段粗搜定位优质区域，第二阶段在该区域内精搜
- 05 对不重要的参数使用默认值，只对核心超参数进行搜索

## NAS搜索到的架构无法部署

// 快速修复

加入延迟约束 + 硬件感知搜索 + 候选架构筛选

// 现象表现

- × 搜索到的架构精度很高但推理速度太慢，无法满足线上延迟要求
- × 参数量太大，无法部署到移动端或边缘设备
- × 模型结构太特殊，目标推理框架（如TensorRT、NCNN）不支持
- × 搜索时只优化了精度指标，完全忽略了部署约束

// 排查步骤

- 01 在搜索目标中加入延迟、参数量、FLOPs等多目标约束，使用加权或帕累托优化
- 02 采用硬件感知NAS算法（如ProxylessNAS、MobileNetV3搜索方法），直接在目标硬件上测速
- 03 搜索前先定义可部署的算子白名单，排除目标框架不支持的操作
- 04 搜索过程中实时评估候选架构的可部署性，提前筛掉不符合要求的
- 05 搜索完成后，在目标硬件上进行实际推理测试验证

## 过拟合验证集导致泛化能力差

// 快速修复

独立测试集 + 早停 + 交叉验证

// 现象表现

- × 验证集分数越来越高，但测试集性能反而下降
- × AutoML选出的"最佳模型"在真实业务数据上表现很差
- × 搜索试验次数越多，最终模型泛化能力越差
- × 验证集被反复"偷看"，相当于间接参与了训练

// 排查步骤

- 01 保留完全独立的测试集，只在最终评估时使用，绝对不参与搜索过程
- 02 设置合理的试验次数上限和早停机制，避免过度搜索验证集
- 03 采用交叉验证代替单一验证集，减少对特定验证划分的过拟合
- 04 使用正则化强度更高的搜索策略，避免模型过于复杂
- 05 最终模型选择时，优先考虑验证集性能稳定且排名靠前的，而非第一名

## AutoML完全替代人工调参导致可控性差

// 快速修复

人工baseline + AutoML精调 + 关键参数手动控制

// 现象表现

- × 模型效果波动大，每次AutoML运行结果差异明显
- × 不知道哪些参数真正重要，调参全靠黑盒
- × 模型可解释性差，出了问题难以定位原因
- × 业务方对自动搜索的模型不信任，不敢上线

// 排查步骤

- 01 先用人工经验建立强baseline，明确哪些参数是核心敏感的
- 02 AutoML只调不确定的、非核心的参数，关键参数保留人工控制
- 03 搜索完成后做参数重要性分析，理解各参数对结果的影响
- 04 对AutoML选出的最优参数进行人工合理性校验
- 05 保留人工干预的入口，关键决策由人来做最终确认
