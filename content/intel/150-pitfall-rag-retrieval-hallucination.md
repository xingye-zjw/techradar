## 为什么你要学它

这是开发中非常容易踩的一个坑：**RAG 检索到了但回答不对 / 幻觉严重**。

RAG 系统检索到了相关文档，但 LLM 生成的回答与检索内容不符，甚至凭空捏造事实。这是 RAG 项目中最常见的质量问题，直接影响系统可用性。

如果你正在遇到类似问题，或者想提前预防，这篇卡片会帮你快速定位问题、找到解决方案，并学会从根源上避免它。

## 一句话概览（快速版）

> **快速修复：换中文 embedding + rerank + 明确引用约束的 Prompt**

核心要点：
- **现象**：向量搜索返回了相关文档，但 LLM 回答与文档不符
- **根因**：检索阶段：embedding 模型对中文语义理解不足，或 chunk 切分不合理导致关键信息丢失。生成阶段：Prompt 未明确约束 LLM 只能基于给定上下文
- **解决**：按照下方 6 步标准流程排查

## 核心拆解

### 🔑 典型症状

- × 向量搜索返回了相关文档，但 LLM 回答与文档不符
- × 答案引用了训练知识而非检索文档（幻觉）
- × 中英文混排检索结果差，语义不匹配

### 🔑 根本原因

检索阶段：embedding 模型对中文语义理解不足，或 chunk 切分不合理导致关键信息丢失。生成阶段：Prompt 未明确约束 LLM 只能基于给定上下文回答，导致模型依赖预训练知识。

## 完整排查方案

按照以下步骤逐一排查，通常能在几分钟内定位并解决问题：

01. 为中文任务选用 BGE-M3 / text-embedding-3-large 等中文表现强的 embedding
02. chunk size 取 200-400 token，务必设置 10-20% overlap，避免关键句被切分
03. 用 reranker（bge-reranker-large）对 Top-k 结果二次排序，取 Top-3 给 LLM
04. Prompt 中明确要求：仅基于给定上下文回答，若无法回答则说不知道
05. 可将 query 先让 LLM 改写为更适合检索的标准问题（HyDE 模式）
06. 检查是否向量数据库中真的存入了正确的 chunk（抽查几条 embedding）

### 快速修复（救急用）

如果你现在就卡在这里，先试试这个：

> 换中文 embedding + rerank + 明确引用约束的 Prompt

## 预防措施

知道怎么解决还不够，更重要的是**下次别再踩**：

- 中文任务优先选用 BGE-M3 或 text-embedding-3-large 等中文表现强的 embedding 模型
- chunk 切分时设置 10-20% overlap，避免关键信息被切断
- Prompt 中明确约束：仅基于给定上下文回答，无法回答则说不知道
- 使用 reranker 对检索结果二次排序，只取 Top-3 最相关的文档给 LLM

## 常见误区

1. **只看表面现象，不深挖根因** — 问题暂时解决了，但过段时间又出现
2. **跳过排查步骤直接试方案** — 容易引入新的问题
3. **不做预防措施** — 同一个坑踩好几次

## 推荐学习顺序

1. 先看「典型症状」确认你遇到的是不是这个问题
2. 再看「快速修复」试试能不能立刻解决
3. 如果不行，按照「完整排查方案」一步步来
4. 最后一定要看「预防措施」，避免下次再踩
