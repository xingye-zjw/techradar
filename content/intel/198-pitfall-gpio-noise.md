## 为什么你要学它

这是开发中非常容易踩的一个坑：**GPIO 输入噪声导致按键误触发/计数错误**。

嵌入式系统中几乎所有外部输入都走 GPIO：按键、旋转编码器、限位开关、霍尔传感器、接近开关、流量计脉冲、急停按钮……这些信号从物理世界进入 MCU，一路上会遇到各种干扰：机械按键按下/释放瞬间的金属弹跳（几十 ms 的"抖动"）、电机/PWM 产生的电磁干扰耦合到长导线上、继电器断开时的反向电动势尖峰、开关电源的纹波传导到地平面。结果就是：你按一次按键，系统认为你按了三次；旋转编码器转一格，计数跳了 ±5；流量计脉冲每个真实脉冲都被识别成两个，产量统计翻了一倍；急停按钮因为一次毛刺被误触发，整条产线紧急停机造成巨大损失。GPIO 输入噪声是嵌入式项目里最常见也最容易被低估的问题，"等量产了再处理"最后往往变成"客户现场紧急救火"。

如果你正在做 STM32/ESP32/Arduino 等 MCU 的按键、编码器、脉冲计数等 GPIO 输入项目，或者遇到了"按一下触发多次"、"计数不准"、"偶尔跳一下"等灵异问题，这篇卡片会帮你建立硬件滤波 + 软件消抖 + 时序验证三层完整的抗噪声体系。

## 一句话概览（快速版）

> **快速修复：按键加 10ms 软件消抖定时器 + 中断输入用连续 2 次相同采样才确认有效 + 硬件 RC 低通滤波**

核心要点：

- **现象**：机械按键按一次系统识别为 2-10 次；旋转编码器每转一格计数跳 ±3 或更多；外部脉冲计数每个真实脉冲计数值加 2；以上问题都是"偶尔发生"，不容易稳定复现
- **根因**：机械触点弹跳（Bouncing，几十 ms 期间高低电平跳变几十次）、电磁干扰（EMI 耦合到长导线产生 ns-us 级毛刺）、电源/地噪声叠加到 GPIO 阈值附近（输入电平刚好在 VIL/VIH 临界区域来回跳）
- **解决**：按照下方 6 步标准流程从硬件到软件分层处理

## 核心拆解

### 🔑 典型症状

- × 机械按键按一次系统识别为 2-10 次，菜单切换、UI 翻页、计数累加完全不对
- × 旋转编码器每转一格计数跳 ±3 或更多，长时间累积误差越来越大
- × 外部脉冲计数器（流量计、霍尔传感器）的读数大约是真实值的 1.8-2.2 倍，也就是每个脉冲被识别两次
- × 急停按钮偶尔在没有被按的时候触发紧急停机，但事后怎么按都复现不出来
- × 上述问题有明显的"环境相关性"：旁边电机一启动就更容易出问题，或者深夜安静环境下几乎不出问题

### 🔑 根本原因

机械触点弹跳（Bouncing，几十 ms 期间高低电平跳变几十次）、电磁干扰（EMI 耦合到长导线产生 ns-us 级毛刺）、电源/地噪声叠加到 GPIO 阈值附近（输入电平刚好在 VIL/VIH 临界区域来回跳）。三种根因的时间尺度完全不同：机械抖动是 10-100ms，电源临界振荡是 ms 级，EMI 毛刺是 us-ns 级，所以处理方式也完全不同。最常见的错误是：只加了软件消抖但没开 GPIO 施密特触发器输入，导致临界区域来回振荡；或者硬件加了大电容滤波但没考虑信号上升沿变缓导致非施密特输入额外抖动。更隐蔽的问题是：外部中断直接用上升沿触发（EXTI_RISING），一个上升沿附近有 5 个小毛刺，就进了 5 次中断，而主循环里的 10ms 消抖根本来不及处理中断级的问题。

## 完整排查方案

按照以下步骤逐一排查，通常能把 GPIO 输入误触发率从百分之几降到百万分之一以下：

1.  **先看波形定根因**：示波器接 GPIO 引脚，设置正常触发边沿，调到 500ns-5ms/div 时间尺度，配合示波器的余辉（Persist）模式抓毛刺。确认是机械抖动（10-100ms 多跳变）、电源临界（缓慢穿过阈值多振荡）还是 EMI 毛刺（窄于 1us 的尖峰）。这一步决定了后面的处理方向
2.  **所有 GPIO 输入开施密特触发器**：STM32 配置 GPIO 时把 Mode 设为 GPIO_MODE_IT_RISING_FALLING 时默认就有施密特，但普通输入模式要确认开了 Schmitt Trigger（某些老 MCU 需要单独配置）。施密特输入的迟滞窗口（Typ 200mV）能彻底解决临界振荡问题
3.  **软件消抖**：机械按键/开关类输入，统一加 10-50ms 消抖定时器。方法 A（定时器法）：GPIO 变化时启动一个 20ms 的软件定时器，20ms 后再采样一次，和变化前相同才算有效。方法 B（状态机法）：每 1ms 采样一次 GPIO，连续 N 次相同才改变状态（N=10 就是 10ms 消抖），绝不允许在 GPIO 中断里直接处理按键逻辑
4.  **中断级防抖**：必须用外部中断的场景（如脉冲计数），不要直接在 EXTI 回调函数里计数。方案：EXTI 触发后启动一个 100us-1ms 的硬件定时器（如 TIM2），定时器到期时再采样一次 GPIO，电平仍然相同才累计计数。同时开启中断"冷却时间"：一次中断处理后，下 50us 内忽略后续同引脚中断
5.  **硬件滤波**：长导线（>20cm）和电磁环境恶劣的场景，GPIO 输入加 RC 低通滤波：100Ω 串联电阻 + 100nF 对地电容，截止频率约 16kHz（足够通过正常 1kHz 以下的机械信号，滤掉 MHz 级的 EMI 毛刺）。如果信号频率较高（>10kHz 编码器脉冲），把电容减小到 1nF，或者用专用的施密特触发器芯片（74HC14）做信号调理
6.  **协议级二次确认**：安全相关的输入（急停按钮、安全门开关），除了上述所有措施外，在应用层加超时检测和协议级确认：比如急停信号连续 10ms 采样都有效才触发停机，且停止后必须手动复位才能恢复。符合 SIL 等级的产品还要用双通道冗余输入做交叉比对

### 快速修复（救急用）

如果你现在就卡在这里，先试试这个：

> 把下面的按键消抖状态机和中断级防抖定时器加到你的代码里，立竿见影

```c
#include "stm32f4xx_hal.h"
#include <stdbool.h>

// ============================================================
// 方案 A：按键消抖状态机（每 1ms 调用一次，适合主循环或 1ms 定时器回调）
// ============================================================
#define DEBOUNCE_MS 20  // 消抖时间，机械按键 10-50ms 合适

typedef struct {
    GPIO_TypeDef *port;
    uint16_t      pin;
    uint8_t       stable_state;   // 确认后的稳定状态：0/1
    uint8_t       sample_count;   // 连续相同采样计数
    uint8_t       last_raw;       // 上一次原始采样值
} DebounceKey_t;

bool DebounceKey_Update(DebounceKey_t *key) {
    uint8_t raw = (HAL_GPIO_ReadPin(key->port, key->pin) == GPIO_PIN_SET) ? 1 : 0;
    bool state_changed = false;

    if (raw == key->last_raw) {
        if (key->sample_count < UINT8_MAX) key->sample_count++;
        // 连续 DEBOUNCE_MS 次相同（每 1ms 调一次，20 次就是 20ms）
        if (key->sample_count >= DEBOUNCE_MS) {
            if (raw != key->stable_state) {
                key->stable_state = raw;
                state_changed = true;
            }
        }
    } else {
        key->sample_count = 0;  // 采样变了，重置计数器
        key->last_raw     = raw;
    }
    return state_changed;  // 返回 true 表示确认后的稳定状态变了
}

// ============================================================
// 方案 B：中断级防抖（脉冲计数、编码器等必须用 EXTI 的场景）
// ============================================================
#define EXTI_COOLDOWN_US 100   // 中断冷却时间，同一引脚 100us 内只处理一次

volatile uint32_t g_pulse_count       = 0;
volatile uint32_t g_last_exti_time_us = 0;
TIM_HandleTypeDef htim7;  // 用一个基础定时器做 100us 二次确认

// ❌ 错误做法：EXTI 回调里直接 ++count，一个毛刺算一次
// void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
//     if (GPIO_Pin == GPIO_PIN_0) g_pulse_count++;
// }

// ✅ 正确做法：EXTI 触发后先启动定时器，定时到期后二次采样确认
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin) {
    if (GPIO_Pin == GPIO_PIN_0) {
        uint32_t now = DWT->CYCCNT / (SystemCoreClock / 1000000U);  // 微妙级时间戳
        if ((now - g_last_exti_time_us) < EXTI_COOLDOWN_US) {
            return;  // 冷却时间内忽略，防止连续毛刺
        }
        g_last_exti_time_us = now;
        // 启动一个 100us 的硬件定时器，到期后再确认一次 GPIO
        HAL_TIM_Base_Start_IT(&htim7);
    }
}

// 基础定时器更新回调（100us 后二次采样确认）
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim) {
    if (htim->Instance == TIM7) {
        HAL_TIM_Base_Stop_IT(htim);
        // 再读一次 GPIO，确实还是高电平才算有效脉冲
        if (HAL_GPIO_ReadPin(GPIOA, GPIO_PIN_0) == GPIO_PIN_SET) {
            g_pulse_count++;  // ✅ 二次确认通过，才真的 +1
        }
    }
}
```

## 预防措施

知道怎么解决还不够，更重要的是**下次别再踩**：

- 项目启动阶段的 GPIO 配置表模板：默认所有输入引脚勾选 Schmitt Trigger，按键类引脚备注"需要 20ms 软件消抖"，高速计数类引脚备注"硬件加 RC 100Ω + 100nF"
- PCB 设计阶段：所有外部 GPIO 输入走线靠近 MCU 端放一个 0402/0603 的 100nF 电容和 100Ω 电阻预留位，不用的话 0Ω 也能贴。现场 EMI 严重时直接焊元件，不用改版
- 外部走线规范：输入信号线不要和 PWM、继电器驱动、电源开关等高 dv/dt 线走平行层，必须交叉时走 90° 垂直交叉。长导线建议用屏蔽线，屏蔽层单端接地
- 测试阶段必做项目：静电放电（ESD）枪对 IO 口 ±8kV 接触放电、附近电钻/变频器启动、电源线上注入 100kHz-100MHz 传导干扰，这三个测试过了才叫抗干扰合格
- 量产版本在 HAL_GPIO_EXTI_Callback 里加计数器（正常版本#ifdef 保护），每个 GPIO 的中断次数和采样次数写入寄存器，产测时对比：真实脉冲 10000 个中断次数应该是 10000±5，如果是 18000 次就说明有双重触发问题需要改固件/硬件

## 常见误区

1. **"我加了 HAL_Delay(10) 在按键中断里就叫消抖了"**：中断里绝对不能加 HAL_Delay（阻塞式延时），而且中断本身就是被毛刺触发的，进来后延时 10ms 不能解决"触发了多次中断"的问题
2. **"我硬件上加了个大电容（10uF），应该不用软件消抖了吧"**：大电容只能滤高频，机械抖动本身就是 10-100ms 的低频信号，电容根本滤不掉，只能让边沿变缓反而导致非施密特输入多振荡几次。硬件 + 软件同时处理才是正解
3. **"旋转编码器用外部中断双边沿计数很简单，直接 EXTI 回调 ++/--"**：编码器 A/B 相的边沿跳变附近有大量抖动和亚稳态，直接计数误差率可达 10-20%，必须用 4x 采样状态机（每 100us 采样一次 A+B 组合状态，查表判断方向），误差率可以降到 0.01% 以下

## 推荐学习顺序

1. 先看「典型症状」确认你的 GPIO 问题属于机械抖动、临界振荡还是 EMI 毛刺
2. 再看「快速修复」把消抖状态机和中断级二次确认加进去，80% 的常见问题当场解决
3. 如果现场 EMI 很严重，按照「完整排查方案」的 6 步从示波器抓波形到 RC 硬件滤波一层层加固
4. 最后一定要看「预防措施」，把施密特输入、RC 预留位、ESD 抗扰测试固化到 PCB 设计和量产测试流程中
