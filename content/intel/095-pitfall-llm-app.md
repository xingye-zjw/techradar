## RAG 检索到了但回答不对 / 幻觉严重

// 快速修复

换用 BGE-M3 等中文表现强的 embedding 模型 + reranker 二次排序 + 明确引用约束的 Prompt

// 现象表现

- × 向量搜索返回了相关文档，但 LLM 回答与文档内容不符
- × 答案引用了训练知识而非检索文档
- × 中英文混排时检索结果差，混合语言场景下语义匹配失效
- × 模型一本正经地胡说八道，生成看似合理但实际错误的内容

// 排查步骤

- 01 选用 BGE-M3 等中文表现强的 embedding 模型，中文语义检索优先用国产嵌入模型
- 02 chunk size 取 200-400 token，设置 10-20% overlap，避免信息在 chunk 边界被切断
- 03 用 reranker 二次排序，对初筛结果进行更精准的相关性重排
- 04 Prompt 明确约束模型必须基于检索内容回答，添加"如文档不包含请明确说明"规则
- 05 HyDE 模式改写 query，用小模型生成假设性答案再检索，提升检索召回率
- 06 检查向量数据库 chunk 是否正确入库，验证分词和向量化的完整流程

## 大模型 Prompt 转义字符错误导致 JSON 输出截断

// 快速修复

使用 json.dumps(prompt_content) 或直接使用 Function Calling / JSON Mode 代替手动字符串拼接

// 现象表现

- × LLM 输出的 JSON 在中途被截断，解析时报 Unexpected end of JSON input
- × JSON 解析失败，提示 Illegal quote 或 unterminated string
- × 只在特定 Prompt 内容时出现，字符串含特殊字符时必现
- × 输出格式混乱，JSON 中混入普通文本或 markdown 标记

// 排查步骤

- 01 检查 Prompt 中是否包含未转义的双引号、换行符、制表符等特殊字符
- 02 Python 中使用 json.dumps() 或 re.escape() 对用户输入进行转义处理
- 03 避免在 JSON 模板字符串中直接放入未转义的双引号内容
- 04 使用 Function Calling 的 JSON Mode，让模型直接输出结构化数据而非手动拼接
- 05 对 Prompt 中的变量使用显式转义，确认特殊字符被正确处理

## LLM 输出不稳定 / 重复复读

// 快速修复

增加 temperature + 设置 presence_penalty + 实现输出去重逻辑截断重复片段

// 现象表现

- × 相同 Prompt 多次调用输出差异大，结果不可复现
- × 模型陷入复读循环，输出"......"或重复短语
- × 输出质量不稳定，时好时坏无法保证一致性
- × 长文本输出时后半段质量明显下降或开始复读

// 排查步骤

- 01 合理设置 temperature，事实类任务设 0.0-0.3，创造性任务可设 0.7-1.0
- 02 检查 top_p 和 top_k 参数，top_p 过高会增加随机性
- 03 设置 presence_penalty（存在惩罚）鼓励模型生成新词汇
- 04 设置 frequency_penalty（频率惩罚）减少重复 token 的生成
- 05 实现输出去重逻辑，检测并截断连续重复的片段
- 06 对输出长度进行限制，用 max_tokens 防止过长输出

## LLM API 调用超时 / 频率限制

// 快速修复

添加重试机制 + 指数退避 + 请求队列限流

// 现象表现

- × API 调用报 timeout 错误，连接超时或读取超时
- × 429 Rate Limit 错误，请求频率超出 API 配额
- × 间歇性调用失败，时好时坏不稳定
- × 批量处理时大量请求失败，影响整体任务完成

// 排查步骤

- 01 实现重试逻辑，设置 max_retries 参数，失败后自动重试
- 02 使用指数退避（exponential backoff），每次重试等待时间倍增
- 03 添加请求队列控制 QPS，使用令牌桶或漏桶算法限流
- 04 监控 API 配额使用情况，提前预警避免触发限制
- 05 实现请求批处理，将多个小请求合并减少 API 调用次数
- 06 配置多模型冗余，主模型失败时自动切换到备用模型
